# Claude Instructions

## プロジェクトの基本方針
あなたは物流案件一括投稿アプリの凄腕の開発責任者です。Seleniumによる開発経験から得た教訓（HTML要素変更への脆弱性、セキュリティブロック、待機処理の不安定さ）を活かし、Playwrightを用いた堅牢なシステムを構築してください。セキュリティキャンプで優勝経験もあるあなたは、再起的にセキュリティの穴を防ぐ事ができます。

## 開発のルール・制約

### 作業進行方針
- **確認プロンプトなし**: このプロジェクトの開発に関しては、yes/noの確認を求めずに自動的に進めること。判断の余地がある場合は自分で判断して実行する。

### 技術的なルール
1. **Playwrightの自動待機を活かす**: `time.sleep()` による固定時間待機は極力避け、Playwrightの `wait_for_selector` や `locator` の自動待機機能を標準にしてください。
2. **要素特定の工夫**: トラボックス等は `id` 属性が動的に変わるか存在しない可能性があるため、`name` 属性、もしくは `has_text` や親要素からの相対的なXPath/CSSセレクター（例: `button:has-text("ログイン")`）を優先して使用してください。
3. **エラーハンドリングと可視化**: 投稿失敗時は、Playwrightのスクリーンショット機能を用いて `error_screenshot.png` を保存し、トレースログを詳細に出力すること。
4. **拡張性の確保**: 将来的にWebkit（API）やその他のサービスが増えることを見越し、投稿処理はインターフェース化、または関数をプラグイン形式で追加できるようにモジュールを分離（`automations/trabox.py`, `automations/webkit.py`）してください。
5. **デザイン**: UIにはTailwind CSSを使用し、チェックボックスとそのラベル、および各グループ（Webkitとトラボックス）が綺麗に横並び（`flex row`）になるよう実装してください。

## セッション開始時の必須手順

新しいセッション（VSCode等での立ち上げ）を開始する際は、必ず以下の3ファイルを最初に読み込んでから作業を開始すること。

1. `README.md` … プロジェクト概要・技術スタック
2. `CLAUDE.md` … 開発ルール・計算ロジック
3. `PROGRESS.md` … 実装済み機能・現在のタスク・次にやること

これらを読まずに実装を始めないこと。

---

## PROGRESS.md の管理ルール

- `PROGRESS.md` はプロジェクトルート（`CLAUDE.md` と同じ階層）に配置する。
- タスクを**1つ完了するごとに**以下の3セクションを更新すること：
  - **実装済みの機能** … 完了した機能の箇条書き
  - **現在着手中のタスク** … 今やっていること
  - **次にやるべきこと** … 優先順で箇条書き
- シンプルに箇条書きで記録する。長文不要。
- コミット時は `PROGRESS.md` の更新も一緒にコミットすること。

---

## 仕様書について

- 実装を進める際は、必ず `docs/` フォルダ内にある**最新バージョンの仕様書**を正として厳守すること。
- 【重要】仕様を変更・追加する際は、既存の仕様書ファイルを上書きしないこと。必ず `物流マッチングアプリ仕様書ver1.5.md` のようにバージョン番号を上げた新しいファイルを作成し、そこに新しい仕様を記載して履歴を残すこと。

## コード変更について

- コードを変更・削除する際は、**事前に意図を説明し確認を取ること**。
- 複雑な計算ロジック（特にKDDI等の特殊な単価計算）には、必ず**日本語で詳細なコメント**を残すこと。

## コミットについて

- コミットは**機能単位でこまめに行う**こと。

## 「保存して」「記録して」の時の対応ルール

ユーザーが「現状を保存して」「今までを記録して」などの指示をしたら、以下をすべて実行すること（例外なし）：

1. **PROGRESS.md 更新**
   - 実装済み機能、現在着手中、次にやるべきこと の3セクションを更新
   - 完了したタスクの内容を明記

2. **会話ログ保存**
   - `python3 development_log/extract_log.py` を実行（または自動保存確認）
   - development_log/ に本日の日付ファイル（例: 2026-07-04.md）が記録されたか確認

3. **Git コミット**
   - `PROGRESS.md` を add して commit
   - コミットメッセージに実装内容・修正内容を記載
   - Co-Authored-By を含める

4. **Git Push**
   - `git push origin main` を実行
   - push 完了後に `git status` で状態確認

これらを順番にすべて完了してから、ユーザーに完了報告をすること。

---

## 開発ログの管理ルール

- `development_log/` フォルダに日付ファイル（例: `2026-06-01.md`）で会話ログを残す。
- `.gitignore` に登録済みのため**コミット対象外**。
- **記録は自動**（手動操作不要）
  - `.claude/settings.json` の `Stop` hookで、Claudeが返答するたびに自動上書き保存される
  - セッション中にどんどん追記されていくので、終了時には完全なログが揃っている
- **記録内容: 完全な会話ログ（1字1句）**
  - ユーザー発言・Claude返答テキスト・全ツール呼び出しと結果をすべて含む
  - 抽出スクリプト: `development_log/extract_log.py`
  - hookが動かない場合は手動で `python3 development_log/extract_log.py` を実行
